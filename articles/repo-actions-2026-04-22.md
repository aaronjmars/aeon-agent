3. Add a Smithery badge to `README.md` and `aeon` to GitHub topics; write a one-paragraph listing description to `docs/smithery-listing.md` for reuse.

### 2. Cross-Fork Skill Customization Digest
**Type:** Community
**Effort:** Small (hours)
**Impact:** 33 forks, ~26 active. Fork operators are making individual skill enable/disable decisions (and sometimes writing new skills) with no visibility into what peers are doing. A weekly digest surfaces: which skills forks enable that upstream doesn't, skills forks consistently disable (candidates for defaulting-off upstream), and any fork-unique skill names not present in upstream. This is the missing peer-learning layer between skill-leaderboard (what's popular) and fork-fleet (how different are forks).
**How:**
1. New `skills/fork-skill-digest/SKILL.md` — paginate fork list via `gh api repos/aaronjmars/aeon/forks`, fetch each fork's raw `aeon.yml`, extract `enabled:` field per skill block.
2. Diff each fork's enabled set against upstream's; aggregate across all forks: count forks that enable each skill (when upstream has it disabled), count forks that disable each skill (when upstream has it enabled), surface skill names appearing in fork aeon.yml blocks but absent from upstream.
3. Format as a ranked table (skills with the widest fork/upstream divergence at top), send via `./notify`, log to `memory/topics/fork-skill-digest.md`; register in `aeon.yml` on Sunday cron alongside skill-leaderboard.

### 3. Webhook-to-Skill Bridge
**Type:** Integration
**Effort:** Medium (1–2 days)
**Impact:** Every current skill trigger is either a cron schedule or a manual `workflow_dispatch`. External events — a new GitHub star, a Discord message, a Zapier trigger from a calendar or CRM — can't drive aeon today. A webhook bridge would let any external system POST a payload and have aeon dispatch the right skill. This closes the reactive execution gap and opens aeon to automation platforms (Zapier, Make.com, n8n) where the real operator-facing integrations live.
**How:**
1. Create `skills/webhook-bridge/SKILL.md` — reads a `.pending-webhook/` directory (same pattern as `.pending-notify/` and `.pending-replicate/`), each `.json` file contains `{skill, var, payload}`; dispatches `gh workflow run aeon.yml -f skill=<skill> -f var=<var>` for each and archives processed files.
2. Write `scripts/webhook-receiver.sh` — a minimal bash HTTP handler (using `nc` or a lightweight tool available in the Actions sandbox) OR a Cloudflare Worker template at `docs/webhook-worker-template.js` that forwards payloads to `gh workflow run` via the GitHub API with a PAT. Include sample handlers for: GitHub star event → star-milestone, push event → repo-pulse, custom POST → arbitrary skill.
3. Document in `README.md` under a new "Webhooks" section; add a `WEBHOOK_SECRET` secret to the validated secrets list in `./onboard`.

### 4. Public Agent Status Page
**Type:** DX / Community
**Effort:** Small (hours)
**Impact:** Fork operators and potential adopters currently have no way to see if aeon is healthy without GitHub Actions access. A live `docs/status.md` — auto-generated by heartbeat on every run and committed to the repo — becomes a public trust signal. GitHub Pages renders it as a readable page. Operators forking the repo get this capability for free and can point their communities to it. As community attention grows (Vercel incident, $AEON Twitter activity), external visibility into agent health matters.
**How:**
1. Modify `skills/heartbeat/SKILL.md` — after the standard per-skill status table, write a `docs/status.md` using the same data (last-run timestamps, pass/fail per skill, open issues count from `memory/issues/INDEX.md`); format as a simple Markdown table with an "Updated:" header showing the UTC timestamp.
2. Commit `docs/status.md` via `gh` in the same step (same pattern as other skills that commit output files); add a `.nojekyll` file and GitHub Pages config (`docs/` source) if not already set, so the page renders at the repo's Pages URL.
3. Add a "Status" badge to `README.md` linking to the Pages URL; add `status-page` to `aeon.yml` topics list.

### 5. Skill Run Analytics Widget
**Type:** DX / Performance
**Effort:** Small (hours)
**Impact:** heartbeat gives a binary ok/not-ok per skill. skill-health does a deep audit of one skill at a time. There's no fleet-level performance view — no way to see which skills run most often, which have the highest failure rate, which haven't run in their expected window, or which consume the most tokens. This closes the observability gap and gives operators a weekly fleet-level dashboard entry. Especially timely now that the 80 autoresearch-evolution rewrites added new exit taxonomies (`SKIP_UNCHANGED`, `NEW_INFO`) that the existing health checks may not parse correctly.
**How:**
1. New `skills/skill-analytics/SKILL.md` — runs `./scripts/skill-runs --json --hours 168` (7-day window) to get per-skill run counts, pass rates, failure patterns; augments with skill categories from `generate-skills-json` output.
2. Produce a ranked table sorted by run count desc, with columns: skill name, 7d runs, success rate, last status, anomaly flag (red if: success rate <80%, or zero runs when cron expected). Output to `articles/skill-analytics-YYYY-MM-DD.md` and a JSON spec to `dashboard/outputs/skill-analytics.json` for the json-render dashboard.
3. Register in `aeon.yml` on Wednesday cron (alongside memory-flush); send a summary notification via `./notify` only when at least one anomaly flag is raised (significance-gated, following the autoresearch-evolution pattern from the 80-skill rewrite).
